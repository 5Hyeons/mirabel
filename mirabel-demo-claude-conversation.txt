  TalkMotion UI v2.0 - 내시경 검사 사전교육 플랫폼 프로젝트네요.

  프로젝트 개요

  목적: 100z 미팅 피칭 주간 내용 - 컨텐츠로 된 프로토타입

  주요 프로세스 플로우:
  1. 인증절차 - 메인인 환자인지 확인 (예약번호, 전화번호 등)
  2. 검사 정보 설명 - 내시경 검사 전 안내 (URL 리 이어 전달 포함)
  3. 내시경 교육 방식 선택 - 실시간 vs. 사전 녹화된 콘텐츠
  4. 동의서 작성 - 전자 서명 포함
  5. 상담 챗봇/콜센터 - 추가 문의 처리
  6. AI 상담 or 전문의 연결 - LLMCHAI 통합
  7. 완료 화면

  디자인 특징:
  - 모바일 우선 디자인 (360x600 / iOS: 375x812)
  - 의사 아바타 캐릭터 적극 활용
  - 파란색 주요 컬러 스킴
  - 단계별 프로그레스 인디케이터

  이 디자인은 wallmate-unity-client나 wallmate-manager와 연동될 의료 교육 플랫폼용 UI로 보입니다. 특정 화면이나 컴포넌트를 더 자세히 보고 싶으시면 말씀해주세요!

  작성할 Spec 문서 개요

  1. PROJECT_SPEC.md - 프로젝트 전체 개요

  - 프로젝트 목표 및 비전
  - 주요 기능 요구사항
  - 타겟 사용자 및 사용 시나리오
  - 전체 User Flow 다이어그램
  - 성공 지표 (KPI)

  2. ARCHITECTURE.md - 기술 스택 및 시스템 아키텍처

  - Frontend: React/Next.js, TypeScript, Tailwind CSS
  - Backend: Node.js/Express (Token Server, API Server)
  - WebGL: Unity WebGL Build (LiveKit Unity SDK)
  - AI Agent: Python + LiveKit Agents (STT/LLM/TTS)
  - Infrastructure: 호스팅, CDN, DB 구조
  - 컴포넌트 다이어그램
  - 데이터 플로우 다이어그램
  - URL 생성 및 환자 인증 메커니즘

  3. SCREENS_SPEC.md - 각 화면별 상세 명세

  각 화면마다 다음을 포함:
  - UI 컴포넌트 구조 (Figma 디자인 기반)
  - 상태 관리 (State, Props, Context)
  - 사용자 인터랙션 (버튼, 입력, 제스처)
  - 데이터 바인딩 (API 호출, 로컬 저장)
  - 유효성 검증 (Validation Rules)
  - 에러 처리 (Error States, Fallback UI)
  - 접근성 (ARIA, Keyboard Navigation)
  - 애니메이션 (Transitions, Micro-interactions)

  화면 목록:
  1. 진입화면 (예약 확인)
  2. 의사 소개 화면
  3. 검사 설명 화면 (페이지네이션)
  4. 동의서 - 체크박스
  5. 동의서 - 전자 서명
  6. 동의서 - 음성 녹음
  7. 동의서 완료
  8. AI 상담 화면 (WebGL)

  4. API_SPEC.md - API 및 데이터 구조

  - REST API 엔드포인트
    - GET /api/patient/:token - 환자 정보 조회
    - POST /api/consent/checkbox - 체크박스 동의 저장
    - POST /api/consent/signature - 전자 서명 저장
    - POST /api/consent/voice - 음성 녹음 업로드
    - POST /api/livekit/token - LiveKit JWT 생성
  - 데이터 스키마 (TypeScript Interfaces)
  - 인증 및 보안 (환자 URL 토큰, HTTPS)
  - 에러 코드 정의

  5. LIVEKIT_INTEGRATION.md - WebGL + LiveKit 통합

  - Unity WebGL 빌드 설정
  - LiveKit Unity SDK 통합 방법
  - React ↔ Unity 통신 (JavaScript Bridge)
  - Agent RPC 통신
    - interrupt() - Agent 답변 중단
    - endSession() - 상담 종료
  - 3D 아바타 립싱크 및 애니메이션
  - 음성 권한 요청 및 처리
  - 네트워크 에러 핸들링

> 5번 유효성 검증 과정이 왜 필요해? 

● 좋은 질문입니다! 유효성 검증이 왜 필요한지 구체적인 예시로 설명드리겠습니다.

  ---
  🤔 유효성 검증이 없다면?

  시나리오: 동의서 체크박스 화면

  유효성 검증 없이 코드를 작성하면:
  function ConsentCheckbox() {
    const [agreed, setAgreed] = useState<boolean | null>(null);

    async function handleSubmit() {
      // 검증 없이 바로 API 호출
      await fetch('/api/consent/checkbox', {
        method: 'POST',
        body: JSON.stringify({ agreed })
      });

      router.push('/consent/signature'); // 다음 화면으로
    }

    return (
      <button onClick={handleSubmit}>제출</button>
    );
  }

  문제점:
  1. ❌ 사용자가 체크하지 않고 버튼 클릭 → agreed = null 전송
  2. ❌ 백엔드가 null을 받아서 에러 발생
  3. ❌ 사용자는 "왜 안 되지?" 하고 혼란

  ---
  ✅ 유효성 검증이 있다면?

  import { z } from 'zod';

  // 1. 검증 규칙 정의
  const consentSchema = z.object({
    agreed: z.boolean().refine(
      (val) => val === true,
      { message: '동의 여부를 선택해주세요' }
    )
  });

  function ConsentCheckbox() {
    const [agreed, setAgreed] = useState<boolean | null>(null);

    async function handleSubmit() {
      // 2. 검증 실행
      const result = consentSchema.safeParse({ agreed });

      if (!result.success) {
        // 3. 검증 실패 시 사용자에게 명확한 에러 메시지
        alert(result.error.errors[0].message); // "동의 여부를 선택해주세요"
        return; // 중단!
      }

      // 4. 검증 통과한 경우만 API 호출
      await fetch('/api/consent/checkbox', {
        method: 'POST',
        body: JSON.stringify(result.data) // 검증된 데이터
      });

      router.push('/consent/signature');
    }

    return (
      <>
        <input 
          type="checkbox" 
          onChange={(e) => setAgreed(e.target.checked)}
        />
        <button onClick={handleSubmit}>제출</button>
      </>
    );
  }

  장점:
  1. ✅ 잘못된 데이터가 API로 전송되지 않음
  2. ✅ 사용자에게 즉시 명확한 피드백 제공
  3. ✅ 백엔드 에러 방지 (불필요한 네트워크 요청 차단)

  🎓 요약: 왜 필요한가?

  1. 사용자 경험 개선

  검증 없음: 제출 → 서버 에러 → "뭔가 문제가 있습니다" (???)
  검증 있음: 제출 → "동의 여부를 선택해주세요" (명확!)

  2. 백엔드 보호

  검증 없음: 잘못된 데이터 → 서버 크래시 가능
  검증 있음: 잘못된 데이터 → 프론트에서 차단

  3. 개발자 생산성

  // TypeScript + Zod = 타입 안전
  const result = schema.parse(data);
  // result는 100% 올바른 타입임이 보장됨

> 근데 혹시 이 padding에 border를 포함하는 개념도 있어? 

● 아니요, padding에 border가 포함되지는 않습니다. 하지만 너비(width) 계산에 border를 포함시킬 수는 있습니다!

  📦 CSS Box Model 두 가지 방식

  1. box-sizing: content-box (기본값)

  width = 콘텐츠 영역만

  전체 크기 = width + padding + border

  예시:
  width: 328px;
  padding: 16px;
  border: 1px;

  실제 크기 = 328 + 32(padding) + 2(border) = 362px

  ---
  2. box-sizing: border-box (현재 사용 중!)

  width = 콘텐츠 + padding + border 모두 포함

  전체 크기 = width

  예시:
  width: 328px;
  padding: 16px;
  border: 1px;
  box-sizing: border-box;  ← 이게 핵심!

  실제 크기 = 328px (그대로!)
  콘텐츠 = 328 - 32 - 2 = 294px

  ---
  🔍 현재 코드 확인

  <div className="... box-border ... px-[16px] ... w-[328px]">
                 ↑
           box-sizing: border-box

  box-border (Tailwind) = box-sizing: border-box (CSS)

  동작 방식:

  w-[328px] 지정 시:

  ├─ border-left: 1px    ┐
  ├─ padding-left: 16px  │
  ├─ 콘텐츠: 294px       │ ← 모두 합쳐서 328px
  ├─ padding-right: 16px │
  └─ border-right: 1px   ┘

  ---
  💡 요약

  | 개념                  | border 포함 여부        | 현재 사용      |
  |---------------------|---------------------|------------|
  | padding             | ❌ 포함 안 됨 (별개 개념)    | -          |
  | width (content-box) | ❌ border/padding 제외 | -          |
  | width (border-box)  | ✅ border/padding 포함 | ✅ 현재 사용 중! |

  현재 코드는 box-border를 사용하므로, w-[328px]에 border와 padding이 모두 포함됩니다!
